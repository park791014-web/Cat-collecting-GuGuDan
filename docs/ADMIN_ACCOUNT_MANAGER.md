# 관리자 계정 관리

## 데이터 출처와 계정 식별

- 목록: Firestore `users` 컬렉션
- 업데이트 대상: 사용자 문서 ID. 닉네임이 같아도 문서 ID로 구분한다.
- 닉네임: `nickname`, 없으면 `profile.nickname`, 둘 다 없으면 문서 ID를 표시한다.
- playerId: `playerId` 또는 `profile.playerId`
- 이메일: 사용자 문서에 안정적으로 저장된 구조가 없어 조회하거나 표시하지 않는다.
- 비밀번호와 인증 토큰은 표에 표시하지 않는다.

## 한 계정당 한 줄

각 계정은 하나의 `<tr>`로 렌더링된다. 열은 닉네임, 계정 ID, 코인, 기본권, 고급권, 시즌권, 네 지급 입력, 적용 버튼이다. ID는 한 줄 말줄임과 `title` 전체 확인을 지원한다. 카드 UI는 생성하지 않는다.

## 재화 표시와 반영

게임의 권위 있는 재화 저장소는 사용자 브라우저의 계정별 localStorage다. 로그인 시 현재 재화를 `currencySnapshot`으로 Firestore에 동기화한다. 관리자 표는 마지막 스냅샷과 아직 수령하지 않은 `pendingResources`의 합계를 표시한다. 따라서 사용자가 다른 기기에서 오프라인으로 플레이한 직후에는 마지막 로그인 동기화 시점 값일 수 있다.

지급된 `pendingResources`는 다음 로그인 때 localStorage 재화로 합산되고 대기 값은 0으로 초기화된다. 시즌권은 `seasonTickets.season_01`에 반영되며 시즌 사용자 UI는 계속 비공개다.

## 검색

- 현재 페이지 안에서 닉네임·문서 ID·playerId 부분 검색
- 앞뒤 공백 제거, 한글 및 영문 대소문자 무시
- 현재 페이지에 결과가 없으면 문서 ID 정확 조회 한 번
- Enter, 검색 버튼, 초기화 버튼 지원
- 검색 입력은 최대 80자이며 이미 로딩된 목록 검색은 추가 Firebase 요청을 만들지 않는다.
- 임의 부분 검색을 위해 전체 컬렉션을 다시 읽지 않는다. 전체 닉네임 부분 검색은 별도 검색 인덱스가 없으므로 현재 페이지로 제한된다.

## 페이지네이션

기본 20명이며 20/50/100명을 선택한다. 문서 ID 정렬, `limit(pageSize + 1)`, `startAfter`를 사용한다. 전체 count 쿼리를 수행하지 않으므로 임의의 총인원은 표시하지 않고 현재 범위와 다음 페이지 존재 여부만 표시한다.

## 지급과 중복 방지

- 허용 최대: 코인 1,000,000, 각 티켓 1,000
- 0 이상의 정수만 허용하며 전부 0이면 요청하지 않는다.
- 확인창에 닉네임, 고유 ID, 현재 재화, 지급 수량을 표시한다.
- 클릭 즉시 해당 행 버튼을 잠근다.
- 고유 `actionId`를 만들고 Firestore transaction에서 사용자 존재와 동일 로그 존재를 확인한다.
- 같은 transaction에서 네 `pendingResources`를 원자 증가시키고 `adminGrantHistory/{actionId}` 로그를 기록한다.
- 성공 후 표 수치를 갱신하고 입력을 0으로 초기화한다.

## 오류 처리

목록·검색·지급 오류를 콘솔에 구체적으로 기록하고 화면에는 목록 실패, 사용자 없음, 지급 실패를 구분해 안내한다. 지급 로그는 재화 증가와 같은 transaction이므로 일부만 성공하지 않는다.

## 관리자 권한과 남은 보안 제한

화면은 관리자 로그인 때 생성되는 세션 플래그를 검사하지만, 기존 프로젝트의 관리자 인증 자체가 클라이언트에 하드코딩된 계정/비밀번호 방식이다. 이것은 안전한 서버 관리자 인증이 아니다. 실제 배포에서 일반 사용자의 읽기·쓰기 차단은 반드시 Firebase Authentication의 관리자 claim과 Firestore 보안 규칙으로 강제해야 한다. 서비스 계정 키나 Admin SDK는 프론트엔드에 추가하지 않았다.

## 모바일

표는 최소 너비를 유지하고 표 영역만 가로 스크롤된다. 페이지 전체에 불필요한 가로 스크롤을 만들지 않는다.

## 테스트

닉네임, ID, playerId, 한글, 결과 없음 검색과 음수·소수·상한 초과 거부를 자동 검증했다. 단일 `<tr>`, transaction, 감사 로그, 페이지네이션 코드도 검증했다. 실제 Firebase 쓰기는 운영 데이터 변경을 피하기 위해 자동 테스트에서 수행하지 않았다.
